# Phase 5 report — Agent run lifecycle (API side)

## What works

- **Contract:** `packages/contracts/openapi.yaml` gains a `runs` tag (`POST /tasks/{id}/start-run`, `GET /runs`, `GET /runs/{id}`, `GET /runs/{id}/logs`, `GET /runs/{id}/events/stream`) and an `internal` tag (`POST /internal/v1/runs/{id}/{status,steps,logs}`, `GET /internal/v1/runs/{id}/context`), each internal operation overriding `servers`/`security` to the `/internal/v1` server and a new `serviceToken` (`X-Talos-Internal-Token`) scheme — matches Section 10.2/10.4 verbatim. `packages/contracts/events/{task.run.requested,run.status.changed}.json` — the first two JSON Schemas in that directory, envelope + payload per Section 11.
- **Backend — state machine:** `dev.talos.runs.RunTransitionValidator` — Section 8.2's table encoded as data (happy path CREATED→...→COMPLETED, FAILED from any non-terminal state, CANCELLED from QUEUED..WAITING_APPROVAL only, REJECTED from WAITING_APPROVAL only) plus the per-state timeout `Duration` table. `AgentRun.transitionTo()` (new mutation — first phase this entity needed one), `AgentRunStep.complete()`.
- **Backend — events:** `dev.talos.events` (new module) — `EventEnvelope<T>` (snake_case wire format via `@JsonProperty`), `RabbitConfig` (`talos.events` topic exchange, `JacksonJsonMessageConverter` — Jackson 3-native, not the legacy `Jackson2JsonMessageConverter`), `EventPublisher.publish(eventType, payload)` (routing key == event type).
- **Backend — run service:** `dev.talos.runs.RunService` centers on one method, `transitionRun()`, that every status change flows through (`start-run`'s CREATED→QUEUED, the internal `/status` endpoint, and the reaper): validates the transition, sets `timeout_at` from the per-state duration table, records a `run.status.changed` audit event, applies Section 8.2's task↔run status mapping (see Deviations for the exact interpretation), and publishes `run.status.changed` to RabbitMQ and to the SSE broadcaster. `startRun()` additionally resolves `agentKey`/`authMode` defaults from the project's active `talos.yaml` (`agents.preferred`, falling back to `"claude-code"` per the schema's declared default if the block exists but doesn't name one, or `422 NO_ACTIVE_CONFIG` if there's no active config and no explicit `agentKey`), enforces the `409 ACTIVE_RUN_EXISTS` check, and publishes `task.run.requested`.
- **Backend — public API:** `dev.talos.tasks.TaskController` gains `POST /{id}/start-run`. New `dev.talos.runs.RunController`: `GET /runs` (filtered/paged `RunSummary`), `GET /runs/{id}` (`Run` + `steps`), `GET /runs/{id}/logs` (`?afterSequence=&size=`), `GET /runs/{id}/events/stream` (SSE, `Last-Event-ID` reconnect support).
- **Backend — internal API:** `dev.talos.auth.InternalTokenAuthenticationFilter` + a second `@Order(1)` `SecurityFilterChain` scoped to `/internal/v1/**` in `SecurityConfig` (service-token auth, entirely separate from the JWT chain). New `dev.talos.runs.InternalRunController`: `POST /status` (`transitionRun`, 422 on illegal + audit even on rejection — Section 8.2's explicit instruction, unlike Phase 4's task moves), `POST /steps` (starts a new `RUNNING` step, or completes the most recent open step of that type — one row per step lifecycle, not one row per call), `POST /logs` (persists then relays), `GET /context` (run + task + project + active parsed config, for the Phase 6 orchestrator to bootstrap from).
- **Backend — SSE:** `dev.talos.runs.RunEventBroadcaster` — publishes every log/status/step event to Redis channel `talos:run:{id}:logs` (Section 10.3's exact channel name, reused for all three event kinds with a `{type, data}` envelope) via a `RedisMessageListenerContainer`; `GET .../events/stream` backfills unread `agent_run_logs` rows from Postgres before subscribing live; 15 s heartbeat comments via `@Scheduled`.
- **Backend — reaper:** `dev.talos.runs.RunReaper`, `@Scheduled(fixedDelay = 60000)`, fails every run whose `timeout_at` has passed via the same `transitionRun()` path (so it gets audited, mapped to the task, and published like any other transition).
- **Backend tests:** `RunTransitionValidatorTest` (exhaustive 12×12 matrix + explicit happy-path/skip/terminal/cancel-boundary cases + timeout table). `RunControllerIntegrationTest` (13 cases: start-run success/config-default/no-config-422/409-active-run, the full CREATED→...→COMPLETED walk asserting the task-side mapping at each stage, FAILED→BLOCKED, step start-then-complete updates one row, log ingest→read-back, internal context, service-token auth (missing/wrong/correct), list/get/404). `EventPublisherIntegrationTest` (Testcontainers RabbitMQ — consumes the real message off a test-declared queue and validates it against the JSON Schema files for both event types). `RunEventStreamIntegrationTest` (real running server + raw `java.net.http.HttpClient` streaming read — a log POSTed internally arrives on an open SSE connection; a reconnect with `afterSequence` backfills only what's newer). `RunReaperTest` (calls the reaper directly rather than waiting 60 s; expires a QUEUED run → FAILED + task BLOCKED; leaves not-yet-expired and `WAITING_APPROVAL` runs alone). 72 backend tests total now, all green against Testcontainers Postgres+RabbitMQ+Redis.
- Verified the entire flow live against the real stack: login → create project → sync a `talos.yaml` with `agents.preferred: custom-shell` → create task → `start-run` with no body (defaulted `agentKey` from the config, confirmed) → task auto-moved to RUNNING → internal `GET /context` (run+task+project+activeConfig) → internal `POST /logs` (204) → `GET /runs/{id}/logs` (persisted) → internal illegal transition QUEUED→COMPLETED (422, audited) → internal endpoint with no token (401) → internal legal transition QUEUED→PREPARING_WORKSPACE (200) → confirmed 3 messages landed on the real `talos.events` exchange via the RabbitMQ management API → confirmed all 5 new audit event types (`run.created`, `run.status.changed`, `task.status.changed`, `run.transition.rejected`, and the illegal one surviving its own rolled-back transaction) via `psql`. See Verification.

A bare "set status to CANCELLED" without that would be misleading.
- `GET /api/v1/projects/{id}/runs` (Section 10.2) — still deferred, as flagged in the Phase 3 report; `GET /api/v1/runs?projectId=` already covers the same use case and Phase 5's task list doesn't call for the separate endpoint.

The board's RUNNING/REVIEW/DONE/BLOCKED columns (Phase 4) will start populating once something calls `start-run`, but there's still no "start run" button anywhere in `apps/web` — natural fit for Phase 6, once there's a real pipeline for it to kick off.

## Deviations from the plan

- **Section 8.2's task↔run status mapping ("run `QUEUED`/`RUNNING_*` ⇒ task `RUNNING`") is read as covering every orchestrator-driven intermediate state, not literally just `QUEUED`/`RUNNING_AGENT`/`RUNNING_TESTS`.** The literal table leaves `PREPARING_WORKSPACE` and `REVIEWING` unmapped, which would make the task's board column go stale mid-run for no apparent reason. `applyTaskMapping()` maps `{QUEUED, PREPARING_WORKSPACE, RUNNING_AGENT, RUNNING_TESTS, REVIEWING}` → `RUNNING` and folds `APPROVED` into `WAITING_APPROVAL`'s `REVIEW` mapping (both are "not done yet, still finalizing"), so every non-terminal run status has an explicit task-side mapping and the switch is exhaustive with no silently-dropped case. Flagged as a judgment call, same category as Phase 4's manual-transition restriction.
- **`CANCELLED`'s "task returns to its status before the run" is implemented as unconditionally restoring `READY`, not the task's actual prior status.** Section 9.2's `tasks` table has no column to hold "status before this run" and the plan says not to invent schema. Given Phase 4's `TaskTransitionValidator` only ever lets a human manually move a task into `READY` before `RUNNING` is reachable at all (there's no other path into `RUNNING`), `READY` *is* the actual prior status in the invariant case; this only under-restores if some future phase adds another path into `RUNNING`. Same treatment as `REJECTED`'s target, which the plan states explicitly as `READY`.
- **`AuditService.record()` changed from default (`REQUIRES`) to `REQUIRES_NEW` transaction propagation.** Found via `internalStatus_illegalTransition_returns422AndWritesAuditEvent` failing: the audit row for a rejected transition was being recorded *inside* the same `@Transactional` method that then throws to produce the 422 — Spring's default rollback-on-unchecked-exception rolled back the audit insert along with everything else, silently losing exactly the row Section 8.2 says must exist ("rejects illegal ones with HTTP 422 plus an audit event"). `REQUIRES_NEW` makes every audit row commit independently of its caller's outcome, which is the correct general semantics for a "write-only audit trail" per Section 12.2, not just a Phase 5-specific patch. Re-ran the full existing suite after the change; no regressions.
- **Internal endpoints are documented in the same `openapi.yaml`** (matching that file's own header, written in Phase 3, describing it as covering both `/api/v1` and `/internal/v1`) via per-operation `servers`/`security` overrides rather than a second spec file. The generated Angular client picks up an unused `InternalService` as a side effect — harmless, since nothing in `apps/web` calls it, and regenerating was still necessary to prove the YAML itself parses.
- **`RunEventBroadcaster` reuses one Redis channel (`talos:run:{id}:logs`) for all three SSE event kinds** (log/status/step), wrapped in a `{type, data}` envelope, rather than inventing separate channel names — Section 10.3 only names that one channel, for logs specifically; extending it to carry status/step too (instead of publishing status/step in-process only, which would silently break the "Redis holds... log pub/sub" contract from Section 11 the moment `talos-api` is ever scaled to more than one instance) seemed truer to the architecture than either alternative.
- **`testcontainers-rabbitmq:2.0.5`** added as a test-only dependency (matches the existing `testcontainers-postgresql`/`testcontainers-junit-jupiter` pinning rationale — `spring-boot-testcontainers` doesn't manage bare Testcontainers module versions). A `copyEventSchemas` Gradle task mirrors Phase 3's `copyTalosSchema` pattern (dedicated `generated-resources` dir, not a bare `include()` on an existing srcDir) to get `packages/contracts/events/*.json` onto the *test* classpath for `EventPublisherIntegrationTest`'s schema validation.

None of these required stopping to ask — each is an implementation-mechanism choice or a documented judgment call within an already-specified contract, not a change to any endpoint, schema, or enum the plan defines.

## Notes

- **The SSE integration test's teardown routinely takes the full 30 s Spring Boot graceful-shutdown grace period** (`Shutdown phase ... ends with 1 bean still running after timeout of 30000ms: [webServerGracefulShutdown]`, then "Graceful shutdown aborted with one or more requests still active"). This is expected, not a bug: closing the client-side `Stream<String>` doesn't make the server *immediately* notice the socket is gone — it only finds out on the next write attempt (the 15 s heartbeat), and by then the test's `@SpringBootTest` context may already be mid-shutdown. Cosmetic slowdown only (adds ~30 s to that one test class), not a failure; not worth engineering around given it doesn't affect correctness or CI exit codes.
- **`AgentRunStep`'s "start then complete" semantics (one row per step lifecycle) were an inference, not stated in Section 10.4.** The endpoint contract is just `{stepType, status, summary?}` with no id in the request; picking "find the most recent open (`RUNNING`) row of that `stepType` for this run and complete it, or start a new one" was the only reading that doesn't produce a new orphaned row every time the orchestrator reports progress on the same logical step.
- Caught the `AuditService` propagation bug (see Deviations) via a failing integration test, not via manual verification — the opposite of Phase 3's two bugs, which manual `curl` testing caught and the test suite missed. Worth keeping both kinds of verification in the routine.
