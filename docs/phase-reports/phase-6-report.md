# Phase 6 report — Orchestrator and runner supervisor (dummy flow)

## What works

- **Contract:** `packages/contracts/openapi.yaml` gains `POST /api/v1/runs/{id}/cancel` and `POST /internal/v1/runs/{id}/changes`, plus optional `testStatus`/`workspacePath`/`branchName`/`prompt`/`summary`/`exitCode` fields on `InternalStatusRequest` (see Deviations). New `packages/contracts/runner-api.yaml` (Section 10.5) — the runner supervisor's own OpenAPI file: `GET /health`, `POST /workspaces/prepare`, `POST /runs/{id}/{execute,tests,diff,stop}`, `POST /workspaces/cleanup`. New event schema `packages/contracts/events/run.cancel.requested.json` (see Deviations).
- **`packages/agent-adapter-spec`:** `adapter.py` — the Section 7.1 `AgentAdapter` ABC and dataclasses (`AgentEventType`, `ProviderCapabilities`, `AgentSessionRequest`, `AgentEvent`, `AgentResult`), transcribed verbatim. `custom_shell.py` — `CustomShellAdapter`, a real, fully working implementation: spawns `request.prompt` as a shell command in `workspace_path` under its own process group, streams stdout/stderr lines as `LOG` events (masking every `request.env` value before emission), enforces `timeout_seconds` (kills and reports `success=False` on timeout), `stop()` SIGTERMs the process group then SIGKILLs survivors after 10s, writes its transcript under `provider_home` (never inside the worktree, so it can't pollute the run's own git diff). `claude_code.py`/`opencode.py`/`codex_cli.py`/`openhands.py`/`gemini_cli.py` — `NotImplementedError` stub classes for every adapter in Section 7.4's table, each naming its assigned phase (Section 7.4: "exists only as a stub class"). `registry.py` — `ADAPTERS` dict + `get_adapter_class()`. `tests/contract/test_custom_shell_contract.py` — the Section 7.2 six-point suite run against `CustomShellAdapter` (events yielded, timeout→`success=False`, `stop()` leaves no process-group members, `exit_code` matches, writes nothing outside `workspace_path`/`provider_home`, env values never leak into events).
- **`apps/runner-supervisor`:** FastAPI app (`app.py`) implementing every `runner-api.yaml` endpoint. `workspace.py` — clone-or-fetch the primary repo, `git worktree add -B agent/task-<task-id>-<slug>`, copy filter (`.env*`/`*.pem`/`id_*` built-in + `talos.yaml context.ignore_paths`) applied to the fresh worktree post-checkout. `diff_capture.py` — intent-to-add untracked files only (not `git add -A`, which would stage — and thereby hide from an unstaged diff — file *deletions*; see Deviations), `git diff --numstat`/`--name-status`/plain diff parsed into per-file `ADDED`/`MODIFIED`/`DELETED`/`RENAMED` + counts, `diff.patch` written under `runs/<id>/artifacts/`. `execute.py` + `run_registry.py` — an in-memory `run_id -> AgentAdapter` registry (the runner supervisor has no database), streams `AgentEvent`/`AgentResult` as newline-delimited JSON. `test_command.py` — runs `talos.yaml commands.test` (or short-circuits `exitCode 0` if unconfigured) the same way. `process_utils.py` — shared kill-tree helper for `/stop`. `/workspaces/cleanup` — takes explicit `runIds` from the caller (see Deviations on why no blind age-only sweep). 16 tests: real local git-repo fixtures for workspace/diff, `TestClient` end-to-end for execute→diff, concurrent-run 409, stop-kills-process (verified via `os.killpg` polling), cleanup age/no-op cases.
- **`apps/orchestrator`:** `api_client.py` (`/internal/v1` httpx client), `runner_client.py` (runner-api.yaml httpx client, async-generator streaming for `execute`/`tests`). `locks.py` — `RunLock`, Redis `SET NX EX` + a Lua compare-and-delete release (never releases a lock a later run now owns). `adapters.py` — capability lookup only; actual adapter execution happens inside the runner supervisor, per Section 7's "adapters ... execute inside the runner supervisor." `log_batcher.py` — batches ingest at 50 lines or 2s per the Log flow narrative. `pipeline.py` — `RunPipeline`, driving Section 8.1/8.2 end to end: crash-recovery poisoning check (a redelivered message for a run not in `QUEUED` → `FAILED ORPHANED_BY_RESTART`, no lock touched) → Redis lock (`FAILED CONCURRENT_RUN` on contention) → `PREPARING_WORKSPACE` → `RUNNING_AGENT` → `RUNNING_TESTS` → `REVIEWING` (diff capture → `POST .../changes`) → `WAITING_APPROVAL`; any exception fails the run and releases the lock, never re-raising into the queue (Section 6.3: "the message is not retried"). `handle_cancel_requested()` just forwards to `runner_client.stop()` — no re-transition, since the API already set `CANCELLED` before publishing the event. `main.py` — `aio-pika` bootstrap: declares `talos.events` + a `talos.events.dlx` dead-letter exchange + `talos.dlq`, two quorum queues (`talos.orchestrator.run-requests`, `talos.orchestrator.cancellations`) with `x-delivery-limit: 3` + `x-dead-letter-exchange` (Section 11's "retried up to 3 redeliveries, then dead-lettered" as a declarative RabbitMQ policy, no manual retry-count code), prefetch 1, Redis-cached `event_id` idempotency (24h TTL). 15 tests: `LogBatcher` size/interval/manual-flush, `_handle`'s idempotency skip, and `RunPipeline` against hand-written fakes (happy path status sequence, lock contention, agent failure, poisoned run, no-test-command→`NOT_RUN`, cancel forwarding, and a race-condition regression test — see Verification).
- **`infra/docker-compose.dev.yml`:** `orchestrator` and `runner-supervisor` services added, with `talos_workspaces`/`talos_provider_homes` named volumes. Fixed two real container-startup bugs found only by actually running the stack (see Deviations/Verification).
- **`scripts/smoke.sh`:** the Section 17 end-to-end smoke script — creates a real git fixture repo inside the running `talos-runner-supervisor` container, logs in, creates a project pointing at `file://` that fixture, creates a task whose description is a `custom-shell` dummy command, starts a run, polls to `WAITING_APPROVAL`, and asserts steps/logs/`git_changes` (queried directly via `docker exec talos-postgres psql`) are all populated. Wired into CI as a new `smoke` job (`.github/workflows/ci.yml`), gated behind `docker-build`, with a log dump on failure and unconditional teardown.

## Deviations from the plan

- **`run.cancel.requested` is a new event/routing-key pair not in Section 11's table.** `POST /api/v1/runs/{id}/cancel` (already listed in Section 10.2) needs a way to tell a *live* orchestrator process to kill a run's process tree, and Section 6.3 is explicit that "the orchestrator never receives HTTP calls from the API" — so RabbitMQ is the only channel available. Modeled as a direct structural mirror of the existing `approval.decided` row (same producer→consumer direction and purpose class: an API/human decision that must reach an in-flight run), on the same `talos.events` exchange, consumed by a new `talos.orchestrator.cancellations` queue that parallels the plan's own `talos.orchestrator.approvals`. The API transitions the run to `CANCELLED` (a legal, already-implemented Section 8.2 edge) *before* publishing the event — the orchestrator's job on receipt is purely to call `runner_client.stop()`, never to re-transition. Full JSON Schema at `packages/contracts/events/run.cancel.requested.json`.
- **`InternalStatusRequest` gained six optional fields (`testStatus`, `workspacePath`, `branchName`, `prompt`, `summary`, `exitCode`) instead of new endpoints.** `agent_runs` has had these columns since Phase 2, but Section 10.4 never assigns any endpoint to write them, and Phase 5 (correctly, given its API-side-only scope) left them all unpopulated. The pipeline naturally produces each one at the exact moment it's already calling `/status` to report a transition (workspace path/branch right after `PREPARING_WORKSPACE`→`RUNNING_AGENT`, prompt/exit code after `RUNNING_AGENT`→`RUNNING_TESTS`, test status after `RUNNING_TESTS`→`REVIEWING`), so bundling them onto that one call avoids inventing five new single-purpose endpoints for an already-enumerated list (Section 10.4). All six are optional and additive; every existing caller of `/status` is unaffected.
- **`git add -A -N` does not correctly surface deletions in an unstaged diff.** Discovered by a failing test, not by inspection: `-A` stages *everything*, including deletions — once a deletion is staged, the index and working tree agree the file is gone, so plain `git diff` (unstaged) shows nothing for it. Fixed by only `git add -N` the specific untracked files (`git ls-files --others --exclude-standard`), leaving tracked modifications/deletions to plain `git diff`'s native (no-staging-needed) behavior.
- **`/workspaces/cleanup` takes explicit `runIds` from its caller rather than sweeping by age.** The runner supervisor has no database and cannot itself determine which runs are terminal or whether their branch has an open PR (the two other conditions Section 8.3 requires before deletion) — a blind age-based sweep could delete an in-progress run's workspace. The endpoint's own safety net is purely "only delete a *named* id, and only if it's also older than `maxAgeDays`"; nothing schedules a call to it yet (flagged under Known blockers below, not silently dropped).
- **Two container-startup bugs, found only by actually running `docker compose up` (not by any test suite):** (1) both Python Dockerfiles' `ENTRYPOINT` was `uv run <script>` with no sync-suppression flag; at container start (as the non-root `talos` user) `uv run` tried to reconcile the environment against the *full* dependency set (including the `--no-dev`-excluded dev group), hit `Permission denied` writing into the root-owned build-time `.venv`, and crashed-looped. Fixed with `uv run --no-sync` (skip environment reconciliation entirely at runtime; the image already synced everything it needs at build time). `--frozen --no-dev` was tried first and *looked* like the same fix but wasn't — it still let `uv run` rebuild the local project's own wheel every start. (2) `talos_workspaces`/`talos_provider_homes` are named volumes; Docker seeds a **fresh** named volume from whatever's already at that path in the image, ownership included — but nothing in `runner-supervisor`'s Dockerfile had ever created `/var/talos/{workspaces,provider-homes}`, so the volume was seeded as `root`-owned and the non-root `talos` user got `PermissionError` on its first `git clone`. Fixed by `mkdir` + `chown talos:talos` those paths before `USER talos`. Both bugs are why `docker compose -f infra/docker-compose.dev.yml up` (Hard constraint 10) matters as a literal, not just figurative, gate — neither would have been caught by any unit or MockMvc-style test.
- **A live cancel-mid-run test surfaced a real race the mocked pipeline tests couldn't model:** `/cancel` moves a run to `CANCELLED` (terminal) while the agent is still executing; `runner_client.stop()` then makes `CustomShellAdapter` report `success=False`, which `_run()` correctly turns into an exception — but the pipeline's own cleanup path then tried to report `FAILED` on a run the API had *already* correctly marked `CANCELLED`, and the API's (correct, pre-existing) terminal-state guard rejected that with 422, which the orchestrator didn't catch, producing an unhandled-task-exception log spray (though the run's DB state stayed correctly `CANCELLED` throughout — nothing was actually broken except the log noise and an uncaught exception in the message handler). Fixed by making the pipeline's failure-reporting call best-effort (catch-and-log, not propagate) — reporting `FAILED` is advisory once we're already in the exception-cleanup path; the API's own state is authoritative regardless. Added a regression test (`test_agent_failure_after_run_already_cancelled_does_not_raise`) reproducing the exact fake sequence, and reverified live.

None of these required stopping to ask — the `run.cancel.requested` event is the only one that reads as extending Section 11's table rather than just filling in an already-specified line item, and it does so by mirroring an existing row's shape rather than inventing new architecture; documented here per CLAUDE.md's sign-off requirement.

## Notes

- **`CustomShellAdapter`'s interim Phase 6 semantics: `request.prompt` is treated as a literal shell command, not an assembled AI prompt.** Section 7.3's real prompt assembler is explicitly Phase 7 work; until then, the orchestrator passes the task's `description` straight through as `prompt`. This is consistent with the adapter's own description ("No AI... reused for build/test/deploy steps") and is exactly what `scripts/smoke.sh` exploits (`description: "echo 'hello from talos' | tee SMOKE_TEST.txt"`).
- **Two genuine bugs (the Dockerfile `uv run` crash-loop and the volume-ownership `PermissionError`) were only found by literally running `docker compose up`, and one race condition (the cancel-vs-report-FAILED conflict) was only found by manually driving a live cancel against the real stack** — none of the 38 new automated tests across four modules would have caught any of the three, since all three are specifically about real OS/container/process behavior that mocks and fakes don't model. This reinforces the same lesson Phase 5's report ended on: keep doing both kinds of verification.
- The runner supervisor's diff/numstat parser handles simple renames (`R100\told\tnew` in `--name-status`, `{old => new}`/`old => new` path forms in `--numstat`) but wasn't stress-tested against partial-path renames or copies; adequate for Phase 6's dummy flow, worth another look whenever a real agent (Phase 7+) starts producing more elaborate diffs.
